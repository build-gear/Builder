# Security Policy

## Supported Version

Builder Gear is pre-1.0. Security fixes target the current `main` branch until a stable channel is published.

## Security Boundary

Builder Gear is local-first and uses the installed Codex CLI for execution. The app detects the Codex auth file path and whether it exists, but it must not read, copy, edit, print, upload, or persist auth file contents. The desktop renderer receives auth readiness plus a generic auth-file label instead of the raw auth path. Auth health checks reject symlinked auth files and POSIX auth files with group or other permissions.

Codex runs use `codex exec ... -` and deliver the prompt over stdin instead of argv. This avoids exposing prompt text through dry-run output or process argument inspection while keeping execution inside the audited Codex CLI path. CLI dry-run output and desktop invocation previews redact local executable, workspace, model, and profile path-like values unless an explicitly unsafe private-debug path is selected in the CLI. CLI fatal error output, including parser-level option errors, redacts secret-shaped values and local filesystem paths before writing to stderr.

Runtime output is redacted and capped before it is displayed. CLI, native, local queued-run, and browser-preview event payloads keep workspace selection metadata plus `pathRedacted: true` instead of copying raw workspace paths. Persisted run history stores status summaries instead of stdout, stderr, Codex JSON payload bodies, artifact payloads, prompts, or workspace paths. Diagnostics reports must exclude prompts, event payload bodies, skill instruction bodies, workspace paths, selected workspace folder names, catalog error file paths, and Codex auth contents. Support bundles must keep the same exclusions and redact health-check local paths before writing or printing troubleshooting data. Desktop diagnostics and support-bundle commands write local files, but renderer results expose only file-name labels plus `pathRedacted: true` instead of raw app-data paths.

Core health reports also redact local workspace and Codex auth paths by default, so new CLI or GUI surfaces using `runHealthCheck` start from shareable diagnostics. Explicit unsafe path output is reserved for private debugging paths only.

Desktop UI recovery messages, stack details, and render-failure console logs are sanitized before use. They redact common token/key shapes and replace local filesystem paths with generic labels so crash screenshots and logs do not expose project or account paths. Recovery-screen Reset local state requires confirmation before deleting Builder Gear browser-state keys. Browser-preview fallbacks must not mask native runtime failures: if the Tauri bridge is present but a command fails, the renderer surfaces the failure instead of using sample preview catalog, scheduler, health, diagnostics, or support-bundle data.

Skill manifests are validated before loading or saving. Skill IDs are restricted to path-safe characters, and instruction files must remain inside their skill directory after resolving existing intermediate directories, so workspace skill metadata cannot point Builder Gear at arbitrary local files through symlinked parents.

Workspace roots selected through the CLI or desktop must be real directories, not symlinks. Builder Gear canonicalizes existing workspaces before catalog loading, schedule import, and Codex execution, and it rejects symlinked workspace-owned files or directories such as `skills/`, `ontology/`, `.builder/schedules.json`, and the CLI SQLite schedule database path. Loaded skill manifests must use path-safe IDs and instruction paths that stay inside the skill directory after resolving existing intermediate directories, so catalog reads follow the same workspace boundary as desktop skill writes.

CLI schedule add creates new SQLite schedules immediately, but existing-id updates preview by default and require `--confirm` before replacement. CLI schedule removal previews by default and requires `--confirm` before deleting a persisted SQLite schedule entry. CLI schedule import with `--replace` previews stale SQLite rows by default and requires `--confirm` before deleting them.

Before destructive or overwriting desktop workspace mutations, Builder Gear creates recovery snapshots in `.builder/backups/`. Desktop deletes for schedules, ontology entities, and skills require explicit confirmation before the backed-up mutation runs. Backup creation rejects symlinked source files, symlinked skill directories, and symlinked entries inside skill directories so backup creation cannot be used to copy files from outside the selected workspace. Backup listing, health checks, and diagnostics expose metadata only and reject symlinked backup entries. Desktop and CLI restore accept only a backup basename, infer supported workspace targets from the backup name, and write a `restore-preimage` backup before replacement. CLI restore previews by default and requires `--confirm` before replacement. Desktop and CLI prune keep newest backups by metadata order, preview deletion candidates before destructive pruning, and reject symlinked backup entries before deletion.

The desktop exposes only the streaming, cancellable Codex run IPC path to the renderer. Deprecated synchronous run commands are not registered, keeping long-running execution on the audited event stream with cancellation and persisted-history summarization.

Persisted run history stores summarized event status only and excludes prompts, stdout, stderr, artifact payloads, Codex JSON payload bodies, and workspace paths. Clearing run history from the desktop requires explicit confirmation before persisted summaries are deleted.

Production desktop builds use a CSP that excludes localhost development origins, blocks object embedding and form submission, and limits network access to Tauri IPC. Local Vite/WebSocket development origins are kept in `devCsp` only. The WebView enables Tauri `freezePrototype`, and release metadata checks fail if that hardening or the CSP separation is removed.

## Reporting

Report security issues privately to the project owner before publishing details. Include reproduction steps, affected platform, app version, and whether the issue involves local files, command execution, diagnostics, schedules, or Codex integration.

## Release Requirements

Distribution builds require code signing and notarization. The internal channel may build signed app/package artifacts without updater output. The stable channel must not be published until signed update artifacts and their hosting location are configured.

Run `pnpm release:preflight` before `pnpm release:check:distribution` to verify signing environment variables, local signing tools, macOS Gatekeeper tooling, and keychain identity without printing signing secrets. Platform/channel templates live at `release/{platform}.{channel}.env.example`, and release metadata verifies those templates plus `.github/workflows/release-candidate.yml` stay aligned with `release/distribution-policy.json`. Release commands reject invalid or duplicate platform/channel arguments instead of falling back to defaults. Bundle-producing gates run `pnpm release:smoke-bundle` after Tauri packaging to verify required artifacts, executable bits, macOS `.app` Mach-O link-table readability, Windows MSI/NSIS headers, and Linux AppImage/deb/rpm headers without launching the GUI. The macOS distribution gate must also verify generated `.app` and `.dmg` code signatures, Gatekeeper assessment, and notarization staples before a release manifest is written. Run `pnpm release:check:stable` only for public-channel candidates; it applies the checked-in `release/tauri.stable.conf.json` updater-artifact overlay and additionally requires `TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`, `BUILDER_GEAR_UPDATER_PUBKEY`, and `BUILDER_GEAR_UPDATE_ENDPOINT` with non-placeholder values. Stable builds generate a target-local Tauri config from the updater public key and production HTTPS static JSON feed URL, then pass that generated config to Tauri. Stable updater endpoints must use HTTPS, end in `.json`, and must not contain updater template variables, credentials, URL query strings, fragments, localhost, loopback, private-network, `.local` mDNS hosts, or reserved example hosts. Preflight rejects missing updater private-key files, too-small updater key content, invalid base64 signing material, and Windows certificate base64 that does not decode like a DER PKCS#12/PFX payload. Stable manifests must include updater payloads, generated `.sig` files, and the generated Tauri static updater JSON feed so the update endpoint can publish Tauri-compatible signatures without rebuilding artifacts. Before publishing, run `pnpm release:stage-upload` and `pnpm release:upload-plan`; the plan re-hashes staged files and records the updater feed endpoint path, payload URL, decoded upload paths, and signature artifact without reading or printing signing secrets.

After stable artifacts are published, run `pnpm release:verify-updater -- <stable-manifest>` against the same manifest to compare the hosted HTTPS updater feed with the staged feed. Add `--verify-downloads` for final production checks that reject non-public hosted updater payload URLs before download, then download the hosted updater payload and compare its SHA-256 with the release manifest.

The desktop updater still applies a renderer-side install policy before invoking Tauri's download/install command. It passes the package version into the updater flow and rejects malformed, same-version, and downgrade candidates before prompting the user.

`.github/workflows/release-candidate.yml` provides the hosted manual path for signed candidate builds. It requires an explicit platform/channel dispatch, fails non-`main` and non-single-segment `release/*` refs in a separate guard job before the signing build job starts, scopes signing and updater secrets to the selected platform/channel, validates the selected required secret names before dependency installation without printing values, imports the macOS Developer ID `.p12` into an ephemeral keychain only on macOS runners, restores the default login keychain and deletes the temporary `.p12` plus signing keychain in an `always()` cleanup step, runs the same distribution or stable release gate, verifies the generated manifest, stages only verified manifest/provenance-declared release files, writes the release upload plan, creates GitHub build-provenance attestations for the staged files and plan, and uploads that staged set with bounded retention. `.github/workflows/verify-stable-updater.yml` is the separate production-environment check after stable artifacts are published to the updater host. Its ref guard fails non-`main` and non-single-segment `release/*` dispatches before the production verification job starts. It checks that the selected run is a successful manual `Release Candidate` run from `main` or a single-segment `release/*` branch, records that run's `headSha`, checks out that exact commit for verification, downloads the stable candidate artifact from that run with `actions: read` into an isolated `release-candidate-artifact/` root, grants `attestations: read` only for provenance verification, re-verifies the manifest and upload plan from that artifact root, rejects artifacts whose manifest is not a bundled stable distribution for the requested platform, rejects artifacts whose manifest git commit does not match the selected run, verifies GitHub build-provenance attestations for every staged file in the isolated upload set against the release-candidate signer workflow while rejecting self-hosted attestation runners, compares the hosted updater feed with the staged feed, and can download the hosted updater payload for SHA-256 verification before rollout.

`pnpm release:github-setup -- --repo OWNER/REPO` dry-runs the required GitHub release environments, and `--apply` creates or updates `internal-release` and `production` when admin credentials are available. Setup enables custom deployment branch policies and ensures only `main` plus `release/*` can deploy through those environments. Creating or updating GitHub environments requires repository admin rights. `pnpm release:github-preflight -- --repo OWNER/REPO` then checks environment existence, required secret names, and deployment branch policies before a hosted release candidate is dispatched. JSON preflight reports include remediation commands that name only environments, secret names, and branch policy patterns; they do not include secret values. These commands do not read or print secret values.

`pnpm service:readiness -- --repo OWNER/REPO --stable-manifest <stable-manifest>` is the final local go/no-go audit after release evidence has been generated and stable artifacts have been published to the updater host. It verifies the local release manifest, verifies that the current commit has a successful hosted `CI` workflow run, runs GitHub environment preflight by secret name and deployment branch policy, and verifies the hosted stable updater feed; `--verify-downloads` additionally downloads the hosted updater payload for SHA-256 comparison. Explicit `--skip-github` or `--skip-updater` produces a partial `warn` audit instead of a full production pass. The command also supports `--artifact-root <path>` so downloaded release artifacts can be verified from an isolated root while keeping manifest paths artifact-root-relative.

CI and release candidate workflows disable checkout credential persistence so the GitHub token is not stored in the checked-out repository after the checkout step.

CLI and desktop Codex runs can be bounded with explicit timeout options. A timeout kills the Codex child process, emits a `timedOut: true` error event, and keeps user interrupts as separate cancellation events so operational timeouts are not confused with intentional stops. CLI timeouts return a failing exit code, and CLI SQLite scheduled jobs remain pending instead of being marked complete.

CLI SQLite schedules use an atomic per-schedule `running_started_at` claim before spawning Codex. Overlapping `run-due` or daemon processes sharing the same database skip already-claimed jobs instead of duplicating work. Successful jobs clear the claim while recording `last_run_at`; cancelled, failed, and timed-out jobs clear only the claim so they remain pending. Claims also store the effective timeout from `--run-timeout-seconds`, `runRequest.timeoutSeconds`, or the 24-hour maximum, and claims older than that timeout are treated as stale and can be reclaimed after a crashed process.

Release upload staging copies only files declared in verified provenance and re-hashes each staged copy before upload. Release manifests may declare bundle artifacts only under the expected Tauri bundle output root for the selected platform, mode, and channel; non-bundle readiness manifests must not carry artifacts. The staging path itself must stay inside the repository and must not contain symlinked path segments. Directory artifacts may contain only regular files, directories, and relative symlinks that resolve inside the artifact root; absolute symlinks, symlinks escaping the artifact root, and unsupported filesystem entries fail release verification before upload staging.

Every release candidate must pass `pnpm security:audit`, which fails on low, moderate, high, and critical npm advisories. Release candidates must also pass `pnpm privacy:scan`, which rejects checked-in developer-machine paths, high-confidence secret patterns, private key blocks, and current environment secret values.

Dependency maintenance is part of the release boundary. `.github/dependabot.yml` must monitor npm, Cargo, and GitHub Actions dependencies weekly; release metadata checks fail if the policy is missing, malformed, or stops covering any of those surfaces.

Release candidates also produce a release inventory next to the release manifest. The inventory records hashes for source files, lockfiles, release policy, CI workflow, and generated artifacts so moved or tampered release assets and stale source states can be detected with `pnpm release:verify`. Release metadata verifies `.gitignore` excludes local runtime and build state such as `.builder/`, SQLite queue files, target directories, release-candidate artifact downloads, package tarballs, reports, logs, and coverage output. Release provenance is constrained to manifest-declared artifacts and a fixed operational evidence allowlist, and it rejects local runtime state so evidence files cannot accidentally include workspace data or unrelated build-machine output.
